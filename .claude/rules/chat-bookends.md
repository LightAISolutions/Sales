# Chat Bookends (MANDATORY — EVERY PROMPT)

*Actionable rules for every response. Examples and summary tables in `chat-bookends-reference.md`.*

> **⚠️ PRECEDENCE HEADER — READ FIRST, ENFORCES THE REST OF THIS FILE ⚠️**
>
> The gates in this file (Feature toggle gate, Page Enumeration gate, Response Opener gate) have historically conflicted when toggles suppress visible output but a downstream gate says "EVERY response." The **correct order of operations** is:
>
> 1. **Feature toggle gate (immediately below) always fires FIRST.** It resolves `START_OF_RESPONSE_BLOCK`, `CHAT_BOOKENDS`, `TIMING_ESTIMATES`, and `END_OF_RESPONSE_BLOCK` and determines which downstream work is visible vs silent vs skipped.
> 2. **Response Opener gate (further down) fires SECOND, and is modified by Step 1's resolution.** The Response Opener's Step 1 combined Bash call runs **on every response without exception** — its job is dual: emit the response-start timestamp AND emit the four toggle values from CLAUDE.md so Step 2's branch decision is driven by tool output rather than memory. When `START_OF_RESPONSE_BLOCK` = `Off`, Step 2's opening block is suppressed — proceed directly to Step 3 (work); the timestamp captured in Step 1 is held silently for TIMING_ESTIMATES if that toggle is on. When both `START_OF_RESPONSE_BLOCK` = `Off` and `TIMING_ESTIMATES` = `Off`, Step 2 emits nothing visible and the timestamp goes unused — but Step 1 still runs (the toggle values must be visible in tool output to confirm the both-off branch).
> 3. **Page Enumeration gate fires when a URL section is about to be written**, which is itself gated by `END_OF_RESPONSE_BLOCK` (UNAFFECTED / AFFECTED URLS) and `START_OF_RESPONSE_BLOCK` (PLANNED AFFECTED URLS).
>
> **Every gate below inherits this precedence.** A downstream gate saying "EVERY response, NO EXCEPTIONS" applies only within the visibility regime Step 1 resolved — it is never grounds to emit a bookend that the Feature toggle gate just suppressed. When in doubt, the toggle wins. **Canonical execution point**: Response Opener Step 0 (below) executes this header's toggle-resolution at response start; mid-response gates re-honor the header by re-reading the toggles before emitting.

> **Feature toggle gate** — this is the first gate to fire on every response. Check the Template Variables table for all four toggles:
> - If `START_OF_RESPONSE_BLOCK` = `Off`: **skip the entire opening block** — no `🚩🚩CODING PLAN🚩🚩` header, no coding plan bullets, no `🔗✏️PLANNED AFFECTED URLS✏️🔗` section, and no opening marker (`⚡⚡CODING START⚡⚡` / `🔬🔬RESEARCH START🔬🔬` / `⚓⚓HOOK FEEDBACK⚓⚓` / `🔃🔃CONTEXT COMPACTION RECOVERY🔃🔃`). The response starts with Step 1's combined Bash call (which always runs — see the Response Opener Gate); when `TIMING_ESTIMATES` = `On` the captured timestamp is held silently as the response-start timestamp, otherwise it goes unused. Then proceed directly to work. The `📋📋PLAN APPROVED📋📋` marker (which appears mid-response after `ExitPlanMode` returns, not at response start) is governed by `CHAT_BOOKENDS`, not this toggle.
> - If `CHAT_BOOKENDS` = `Off`: **skip all mid-response phase bookends and their coupled `⏱️` duration annotations** — CHECKLIST, RESEARCHING, NEXT PHASE, BLOCKED, VERIFYING, CHANGES PUSHED, AWAITING HOOK, PLAN APPROVED, and all `⏱️` duration lines are suppressed. Work between phases still happens; it just isn't labeled or timed. The hook anticipation logic (running the three git commands) still executes — only its bookend output is suppressed. Phase bookends and their durations are coupled under this toggle because a `⏱️` duration describes the gap between two phase markers and has nothing meaningful to attach to when the markers are suppressed.
> - If `TIMING_ESTIMATES` = `Off`: **skip all start/end timing markers** — ESTIMATED TIME, REVISED ESTIMATED TIME, ACTUAL PLANNING TIME, ACTUAL TOTAL COMPLETION TIME, ESTIMATE CALIBRATED, and PLAN EXECUTION TIME are all suppressed. The Response Opener Step 2 silent-capture branch does not fire (nothing consumes the captured data). When `START_OF_RESPONSE_BLOCK` = `Off` + `TIMING_ESTIMATES` = `Off`, Step 1's combined Bash call still runs (the toggle values must be in tool output to confirm we're in the both-off branch) — but the captured timestamp goes unused. **When `TIMING_ESTIMATES` = `On`** — the top `ESTIMATED TIME` and bottom `ACTUAL TOTAL COMPLETION TIME` markers always travel together (one toggle owns both), so the parenthetical `(estimated Xm)` on the bottom always has a visible sibling at the top of the response. When `START_OF_RESPONSE_BLOCK` = `Off` + `TIMING_ESTIMATES` = `On`, both markers appear standalone (no `CODING START` to attach the top estimate to, and no closing marker to attach the bottom total to when `END_OF_RESPONSE_BLOCK` is also `Off`).
> - If `END_OF_RESPONSE_BLOCK` = `Off`: **skip the entire work-summary block and the closing marker** — no `─────` dividers, no END OF RESPONSE BLOCK header, no UNAFFECTED URLS, AGENTS USED, FILES CHANGED, COMMIT LOG, WORTH NOTING, SUMMARY, TODO, NEW FOLDERS, AFFECTED URLS, CLAUDE TO DEVELOPER, or closing marker (CODING COMPLETE / RESEARCH COMPLETE). Timing markers (when `TIMING_ESTIMATES` = `On`) still emit standalone before the response ends. Mid-response bookends (when `CHAT_BOOKENDS` = `On`) function fully and independently. `END_OF_RESPONSE_BLOCK` pairs symmetrically with `START_OF_RESPONSE_BLOCK` — one owns the opener, the other owns the closer.
> - **Silent capture in the Timing-only branch** — when `TIMING_ESTIMATES` = `On` + `START_OF_RESPONSE_BLOCK` = `Off`, the response-start timestamp + overall estimate must still be captured even though no visible opening marker emits. The full silent-capture procedure (4-step hard gate + 4-question self-check) lives in the **Response Opener Gate's Step 2 Timing-only branch** below — it is the single authoritative source. When `TIMING_ESTIMATES` = `Off`, no silent capture happens (nothing consumes the data). When `START_OF_RESPONSE_BLOCK` = `On` + `TIMING_ESTIMATES` = `On`, no silent capture is needed — the visible CODING START / RESEARCH START marker already captures the timestamp.
> - **Full independence of all four toggles** — zero cross-dependencies. Every feature of `START_OF_RESPONSE_BLOCK` works without the other three; every feature of `CHAT_BOOKENDS` works without the other three; every feature of `TIMING_ESTIMATES` works without the other three; every feature of `END_OF_RESPONSE_BLOCK` works without the other three. All 2⁴ = 16 combinations produce coherent output. No other repo functionality depends on any toggle being in a specific state — they are purely output-presentation controls.
> - When all four are `On` (the legacy default), all rules below apply as written

- **First output — coding plan** (part of START_OF_RESPONSE_BLOCK): for every user prompt that will involve changes, the very first line written to chat must be `🚩🚩CODING PLAN🚩🚩` on its own line, followed by a brief bullet-point list of what will be done in this response, then a **blank line** followed by `⚡⚡CODING START⚡⚡` on its own line to signal work is beginning. The blank line is required to break out of the bullet list context so CODING START renders left-aligned. **Bullets describe what will be done** — one bullet per distinct action (e.g. "Edit CLAUDE.md to add coding plan rule", "Update README.md timestamp"). Keep the plan concise. **Do NOT quote the user's prompt** — the prompt is already visible above in the chat; quoting it verbatim in the first bullet is redundant. The prompt is preserved separately in `repository-information/CHANGELOG.md` for historical reference. **Do NOT include boilerplate reminders** (e.g. "write in small chunks") — the **Incremental Writing** gate in `.claude/rules/behavioral-rules.md` already enforces that specific rule, and other behavioral gates in the same file cover the rest; emitting them as visible bullets is cargo-cult. This is for transparency, not approval — do NOT wait for user confirmation before proceeding. If the response is purely informational with no changes to make, skip the plan and open with `🔬🔬RESEARCH START🔬🔬` directly (instead of CODING START). **CODING PLAN and CODING START / RESEARCH START appear exactly once per response** — never repeat them mid-response. Use `🔄🔄NEXT PHASE🔄🔄` instead (see below). **Gated by `START_OF_RESPONSE_BLOCK`** — skipped entirely when `START_OF_RESPONSE_BLOCK` = `Off`
- **Planned affected URLs** (part of START_OF_RESPONSE_BLOCK): immediately after the coding plan bullets (after the blank line that ends the bullet list, before ESTIMATED TIME), output `🔗✏️PLANNED AFFECTED URLS✏️🔗` followed by the page URLs expected to be affected by this response — using the same label-URL pair format as the end-of-response `🔗✏️AFFECTED URLS✏️🔗` section (backtick-wrapped `✏️` labels, blockquoted URLs). **Show the current (pre-change) version and status** in the label — read the page's `html.version.txt`, determine the status emoji (🟢/🟡/🔴), and append the version in parentheses: `` `✏️🌐🟢 Homepage (v01.13w)` ``. This gives the user clickable links to the files' current state and shows what version the page is at before changes begin. **Best-effort prediction** — base it on the coding plan bullets; if the scope changes mid-work, the final AFFECTED URLS at the end may differ. When no pages are expected to be affected, output the header followed by `> *No URL pages expected to be affected*`. **Skip entirely for RESEARCH START responses** (no plan, no affected URLs). When the coding plan repeats after `📋📋PLAN APPROVED📋📋`, include PLANNED AFFECTED URLS again after those plan bullets (the prediction may be more accurate post-research). **Gated by `START_OF_RESPONSE_BLOCK`** — skipped entirely when `START_OF_RESPONSE_BLOCK` = `Off`
- **Continuation after user interaction**: when `AskUserQuestion` or `ExitPlanMode` returns mid-response (the user answered a question or approved a plan), the response continues but must **NOT** repeat `🚩🚩CODING PLAN🚩🚩`, `⚡⚡CODING START⚡⚡`, or `🔬🔬RESEARCH START🔬🔬`. Instead:
  - After `AskUserQuestion`: use `🔄🔄NEXT PHASE🔄🔄` with a description incorporating the user's choice (e.g. "User chose option A — proceeding with implementation")
  - After `ExitPlanMode` (plan approved): output `📋📋PLAN APPROVED📋📋` on its own line, followed by `🚩🚩CODING PLAN🚩🚩` with the execution plan bullets, then `⚡⚡CODING START⚡⚡`. This is the **only** scenario where CODING PLAN/CODING START may appear a second time — because plan approval is a distinct boundary between planning and execution, and the user needs to see the execution plan clearly. The `📋📋PLAN APPROVED📋📋` marker signals that this is a continuation, not a new prompt. **Fresh timestamps required** — run `date` once for PLAN APPROVED, then run `date` again for the post-approval CODING PLAN + CODING START pair (a single call covers both, same as the initial opening pair rule). Do NOT reuse the PLAN APPROVED timestamp for CODING START — there is a gap between plan approval and the start of execution (the coding plan bullets and estimated time are written in between), so the timestamps will differ. PLAN EXECUTION TIME is computed from the post-approval start timestamp (whether displayed via CODING START or silently captured), so an inaccurate timestamp here inflates or deflates the reported execution time
- **Checklist running**: output `✔️✔️CHECKLIST✔️✔️` on its own line before executing any mandatory checklist (Session Start, Pre-Commit, Pre-Push), followed by the checklist name (e.g. `Session Start Checklist`). This separates checklist overhead from the user's actual task. Output once per checklist invocation.- **Researching**: output `🔍🔍RESEARCHING🔍🔍` on its own line when entering a research/exploration phase — reading files, searching the codebase, or understanding context before making changes. Skip if going straight to changes without research.- **Mid-response phase marker**: when work within a single response naturally divides into multiple distinct sub-tasks or phases (e.g. "Edit 1" then "Edit 1a: fix related issue"), output `🔄🔄NEXT PHASE🔄🔄` on its own line followed by a brief description of the new phase. **Never repeat** `🚩🚩CODING PLAN🚩🚩`, `⚡⚡CODING START⚡⚡`, or `🔬🔬RESEARCH START🔬🔬` within the same response — those appear exactly once (at the very top). The mid-response marker keeps the top/bottom boundaries of each prompt/response turn unambiguous while still signaling transitions between sub-tasks.- **Blocked**: output `🚧🚧BLOCKED🚧🚧` on its own line when an obstacle is hit (permission denied, merge conflict, ambiguous requirement, failed push, hook check failure). Follow with a brief description of the blocker. This makes problems immediately visible rather than buried in tool output.- **Verifying**: output `🧪🧪VERIFYING🧪🧪` on its own line when entering a verification phase — running git hook checks, confirming no stale references, validating edits post-change. Separates "doing the work" from "checking the work".- **Changes pushed**: output `➡️➡️CHANGES PUSHED➡️➡️` on its own line **immediately** after a successful `git push` completes. This gives the user instant visibility that their changes are on the remote — no need to scroll through tool output to confirm. Follow with a brief note of what was pushed (e.g. "Pushed to `claude/cleanup-xyz` — workflow will auto-merge to main"). This bookend is a timestamped marker like any other — run `date` before writing it. It participates in `⏱️` duration tracking (the previous phase's `⏱️` appears before it, and the next bookend's `⏱️` measures from it)
- **Time estimate**: output `⏳⏳ESTIMATED TIME ≈ Xm⏳⏳` on its own line with a brief reason (e.g. "~8 file edits + commit + push cycle"). Use `Xm` for minutes or `Xs` for sub-minute estimates. **Two contexts**:
  - **Overall** — always appears immediately before `⚡⚡CODING START⚡⚡` or `🔬🔬RESEARCH START🔬🔬`, estimating the full response through the closing marker. Never skipped, even for quick responses. **Plan-mode flows** must include BOTH phases (research/planning + anticipated post-approval execution) — the response is not done at `ExitPlanMode`; it runs through approval and execution until `✅✅CODING COMPLETE✅✅`
  - **Per-phase** — only when a subsequent phase (NEXT PHASE, RESEARCHING, CHECKLIST, etc.) alone will exceed 2 minutes. Skip for quick phases
- The estimate is an annotation, not a bookend — no timestamp, no `date` call, no `⏱️` tracking
- **Heuristics** (sum the expected tool calls and round to the nearest minute):
  - **~10s**: each tool call with payload ≤10 lines (read, edit, grep, glob)
  - **~15s**: each Edit with `new_string` >10 lines; each bash command
  - **~30s**: commit cycle (checklist + staging + commit); push cycle (checklist + push + verify)
  - **~1–2m**: Explore subagent spawn
  - **~3–5m**: Plan subagent spawn (deeper multi-section analysis)
  - **~60–90s**: first-time Playwright install (`pip install` + `playwright install chromium`, ~110MB download)
  - **~30–60s** per Playwright run — but **budget 3–6m total** for layout-sensitive changes (3–5 iterations: first run reveals hidden state, subsequent iterations surface CSS cascade/specificity bugs). Budget the high end for flexbox, grid, aspect-ratio, or any property overridable by a later-in-source rule
- **Discounts** (apply when the pattern fits):
  - **Parallel batching**: a single batch of N independent tool calls counts as ~15s total, not N × 10s. Most push-commit cycles fire 4–6 edits in parallel, so the editing phase is ~15s, not ~50s
  - **Template/consolidation**: work that collapses duplicate content into a pointer to an existing authoritative reference budgets **~50% of the from-scratch estimate** — the canonical target already carries the structure; the consolidation is a trim-and-redirect
- **Revised estimate**: once all initial file reads, greps, and exploration needed to understand the scope of work are complete — regardless of whether a RESEARCHING bookend was used — re-evaluate the overall estimate. If it now differs from the original by ≥1 minute, output `⏳⏳REVISED ESTIMATED TIME ≈ Xm⏳⏳ [HH:MM:SS AM/PM EST]` on its own line with a brief reason (e.g. "~12 files to edit, not 4 as initially expected"). **Run `date` before writing the line** — this is a timestamped marker like any other bookend. Place it inline right after the last read/exploration tool result and before the next action or bookend. **Skip if the original estimate still looks accurate.** The revised estimate replaces the original as the baseline for the ACTUAL TOTAL COMPLETION TIME comparison at the end
- **Actual time**: output `⏳⏳ACTUAL TOTAL COMPLETION TIME: Xm Ys (estimated Xm)⏳⏳` on its own line immediately before the closing marker (`✅✅CODING COMPLETE✅✅` or `🔬🔬RESEARCH COMPLETE🔬🔬`), or standalone before the response ends when `END_OF_RESPONSE_BLOCK` = `Off`. This is the real elapsed wall-clock time from the response start timestamp to the closing marker — computed by subtracting the response start timestamp from the closing marker's timestamp. The response start timestamp is captured by CODING START / RESEARCH START when `START_OF_RESPONSE_BLOCK` = `On`, or silently via `date` at response start when `START_OF_RESPONSE_BLOCK` = `Off` (see feature toggle gate). The parenthetical shows the original overall estimate for easy comparison — because the top ESTIMATED TIME and this bottom total are both governed by the same `TIMING_ESTIMATES` toggle, the parenthetical always has a visible sibling at the top of the response. If a REVISED ESTIMATED TIME was issued, use the revised value instead. **Gated by `TIMING_ESTIMATES`** — emitted only when `TIMING_ESTIMATES` = `On`, skipped entirely when `Off` regardless of the other three toggles. **Not present** before AWAITING USER RESPONSE — that ending uses ACTUAL PLANNING TIME instead. The `date` call for the closing marker (already required) provides the end time — no additional `date` call is needed
- **Plan execution time** (conditional): when a plan approval flow was used (`ExitPlanMode` → `📋📋PLAN APPROVED📋📋`), the timing output must include **both** time markers in this order: (1) `⏳⏳PLAN EXECUTION TIME: Xm Ys (estimated Xm)⏳⏳` — the wall-clock time from the post-approval start timestamp (captured by the post-approval CODING START when `START_OF_RESPONSE_BLOCK` = `On`, or silently when `START_OF_RESPONSE_BLOCK` = `Off`) to CODING COMPLETE, with the post-approval estimate as the parenthetical, then (2) `⏳⏳ACTUAL TOTAL COMPLETION TIME: Xm Ys (estimated Xm)⏳⏳` — the sum of the ACTUAL PLANNING TIME and the PLAN EXECUTION TIME, with the original overall estimate (from the response start) as the parenthetical. **Skip PLAN EXECUTION TIME entirely when no plan approval occurred** — responses without `ExitPlanMode` only show ACTUAL TOTAL COMPLETION TIME as usual. **Gated by `TIMING_ESTIMATES`** — skipped entirely when `TIMING_ESTIMATES` = `Off` along with the other timing markers
- **Estimate calibration**: after computing ACTUAL TOTAL COMPLETION TIME, if the actual and estimated times differ by **more than 5 minutes**, surface the variance via `🔧🔧ESTIMATE CALIBRATED🔧🔧` (positioned after SUMMARY when the end-of-response block is emitted, or standalone before ACTUAL TOTAL COMPLETION TIME when it isn't) with a **brief diagnosis** of the gap — what caused the miss (e.g. "auto-injection of `repo-docs.md` + `changelogs.md` added ~45s of context ingestion not captured in per-tool-call heuristics", or "underestimated the number of file reads — 12 actually needed vs 4 predicted"). **Editing heuristic values in this file is OPTIONAL** — only propose a heuristic change when the gap is clearly explainable by a specific per-tool-call or per-phase number being wrong, and the same pattern would plausibly recur. Most variance is injection-driven or scope-driven and doesn't benefit from heuristic tweaks. **Never make a follow-up commit purely to adjust a heuristic** — if the main commit has already happened, just note the miss and skip the edit. **Skip calibration entirely if the difference is ≤5 minutes** — small-to-moderate variances are normal and noise-bound. **Gated by `TIMING_ESTIMATES`** — skipped entirely when `TIMING_ESTIMATES` = `Off` (there's no estimate to calibrate against when the estimate was never shown)
- **Hook anticipation**: before writing the closing marker (`✅✅CODING COMPLETE✅✅` or `🔬🔬RESEARCH COMPLETE🔬🔬`), check whether the stop hook (`~/.claude/stop-hook-git-check.sh`) will fire. **This check must happen after all actions in the current response are complete** (including any `git push`) — do not predict the pre-action state; check the actual post-action state. **Actually run** the three git commands (do not evaluate mentally): (a) uncommitted changes — `git diff --quiet && git diff --cached --quiet`, (b) untracked files — `git ls-files --others --exclude-standard`, (c) unpushed commits — `git rev-list origin/<branch>..HEAD --count`. If any condition is true, **omit** the closing marker and instead write `🐟🐟AWAITING HOOK🐟🐟` as the last line of the current response — the hook will fire, and the appropriate closing marker (CODING COMPLETE or RESEARCH COMPLETE) should close the hook feedback response instead. **Do not forget the `⏱️` duration annotation** — AWAITING HOOK is a bookend like any other, so the previous phase's `⏱️` must appear immediately before it. After the hook anticipation git commands complete, call `date`, compute the duration since the previous bookend's timestamp, write the `⏱️` line, then write AWAITING HOOK
- **Compaction recovery override** (part of START_OF_RESPONSE_BLOCK): if the conversation context has been compacted (session summary replacing earlier messages), use `🔃🔃CONTEXT COMPACTION RECOVERY🔃🔃` as the first line (with time+date timestamp) instead of any other opening marker. This replaces CODING PLAN, CODING START, RESEARCH START, and HOOK FEEDBACK as the response opener. After the bookend, re-read CLAUDE.md rules (per the Session Start Checklist compaction recovery bullet), then resume the interrupted task — output `🔄🔄NEXT PHASE🔄🔄` to signal resumption and continue working (NEXT PHASE is governed by `CHAT_BOOKENDS`). **Do not re-show reminders** — they were already surfaced earlier in the session before compaction. **Do not redo completed work** — use the compacted summary to understand what was already accomplished and pick up where the interruption occurred. **Gated by `START_OF_RESPONSE_BLOCK`** — skipped entirely when `START_OF_RESPONSE_BLOCK` = `Off`
- **Hook feedback override** (part of START_OF_RESPONSE_BLOCK): if the triggering message is hook feedback (starts with "Stop hook feedback:", "hook feedback:", or contains `<user-prompt-submit-hook>`), use `⚓⚓HOOK FEEDBACK⚓⚓` as the first line instead of `🚩🚩CODING PLAN🚩🚩`, `⚡⚡CODING START⚡⚡`, or `🔬🔬RESEARCH START🔬🔬`. The coding plan (if applicable) follows immediately after `⚓⚓HOOK FEEDBACK⚓⚓`, then `⚡⚡CODING START⚡⚡`. **Gated by `START_OF_RESPONSE_BLOCK`** — skipped entirely when `START_OF_RESPONSE_BLOCK` = `Off`
- **End-of-response sections**: after all work is done, output the sections in the table below **in numeric order**. **Skip the entire block when the response ends with RESEARCH COMPLETE or AWAITING USER RESPONSE** — those endings have no end-of-response block. **The entire block — from the divider through CODING COMPLETE — must be written as one continuous text output with no tool calls in between.** Run `date` for CODING COMPLETE's timestamp **before** starting the block, then output: the last phase's `⏱️` duration, a backtick-wrapped divider `` `─────────────────────────` `` on its own line, `` `END OF RESPONSE BLOCK` `` on the next line, another backtick-wrapped divider on the line after. Then emit the sections below using the pre-fetched timestamp.

  | # | Section | Marker | When included (skip rule) | Content / format |
  |---|---------|--------|---------------------------|------------------|
  | 1 | Unaffected URLs | `🔗🛡️UNAFFECTED URLS🛡️🔗` | Never skipped within the block | Reference URLs + unaffected pages. See Unaffected/Affected URLs bullet below for full rules |
  | 2 | Agents used | `🕵🕵AGENTS USED🕵🕵` | Skip when only Agent 0 (Main) participated | Numbered list: `N. Agent N (Type) — contribution`. Include Agent 0 when any subagent contributed |
  | 3 | Files changed | `📁📁FILES CHANGED📁📁` | Skip if no files were changed | Bullet list with `(edited)` / `(created)` / `(deleted)` tag per file |
  | 4 | Commit log | `📜📜COMMIT LOG📜📜` | Skip if no commits were made | Per commit: `SHA: [SHORT_SHA](commit-url) — commit message`. Full SHA via `git rev-parse`, `ORG/REPO` from `git remote -v`. Plain markdown link — do NOT backtick-wrap the SHA text |
  | 5 | Worth noting | `🔖🔖WORTH NOTING🔖🔖` | Skip if nothing worth noting | Bullet list of attention-worthy observations (e.g. "Push-once already used", "Pre-commit hook re-staged files") |
  | 6 | Summary | `📝📝SUMMARY📝📝` | Never skipped within the block | Concise bullets, each indicating which file(s) were edited. Non-file actions (e.g. "Pushed to remote") need no file path |
  | 7 | To-do list | `📋📋TODO📋📋` | Skip when TODO.md shows `*(No items yet)*` OR when items exist but none were touched this response | `- [x] ~~Item~~` for completed, `- [ ] Item` for pending. See TODO display rules below |
  | 8 | New folders | `📂📂NEW FOLDERS📂📂` | Skip entirely if no new directories were created (no header, no placeholder) | Markdown links: `[folder/path/](https://github.com/ORG/REPO/tree/main/folder/path)` |
  | 9 | Affected URLs | `🔗✏️AFFECTED URLS✏️🔗` | Skip entirely if no pages were affected (no header, no placeholder) | Filtered page list with `✏️` indicator. See Unaffected/Affected URLs bullet below |
  | 10 | Claude to Developer | `💬💬CLAUDE TO DEVELOPER💬💬` | **Only when there's a genuine heads-up or concrete recommendation** — skip the section entirely otherwise (no header, no placeholder). Emitted inside the block for coding responses and outside the block for RESEARCH COMPLETE responses (before ACTUAL TOTAL COMPLETION TIME — see Claude to Developer bullet below) | Heads-up (optional, brief context/caveats) + exactly one **Recommended next step** (primary, Claude's single best judgment — no menu of alternatives) + the exact phrase the user can type to continue with that primary recommendation |

  - **TODO display rules** (when section 7 is emitted): Read TODO.md alongside the `date` call before the block (exception to "no tool calls"). Show the section only when a to-do was **completed** (`- [x] ~~Item~~` crossed off in its original position alongside remaining `- [ ] Item` lines), **added**, or **removed/modified** this response. After the block, edit TODO.md to persist completions (second "no tool calls" exception) — if all items are completed, restore `*(No items yet)*`. **Maximum 10 items** — flag when the user requests an 11th. TODO.md file format is plain `- Item text` lines (no checkboxes in the file; checkboxes are only in the end-of-response display).
  - **Section separation** — a **blank line** must appear between every end-of-response section (after the last content line of one section and before the next section's header). - **Duration annotations (MANDATORY — NEVER SKIP)** — a `⏱️` annotation appears between **every** consecutive pair of bookends, before the end-of-response block, and before any `ExitPlanMode`/`AskUserQuestion` call. **No exceptions** — if two bookends (or a bookend and a user-interaction tool call) appear in sequence, there must be a `⏱️` line between them. **Procedure for EVERY mid-response bookend** (CHECKLIST, RESEARCHING, NEXT PHASE, BLOCKED, VERIFYING, CHANGES PUSHED, AWAITING HOOK, HOOK FEEDBACK, and the end-of-response block): (1) run `date` via Bash, (2) compute the elapsed time since the previous bookend's timestamp, (3) write `⏱️ Xs` (or `Xm Ys` for durations over 60 seconds), (4) THEN write the bookend marker. If a phase lasted less than 1 second, write `⏱️ <1s`. **Self-check**: before writing any bookend, ask "Did I write a `⏱️` line first?" If no, stop and compute the duration before proceeding.
  - **Sub-case — before user interaction**: before calling `ExitPlanMode` or `AskUserQuestion`, in addition to the `⏱️` phase duration, also write `⏳⏳ACTUAL PLANNING TIME: Xm Ys (estimated Xm)⏳⏳` — total time from the response start timestamp to the current moment, compared against the overall estimate. ## Page Enumeration — Mandatory Enumeration Gate

> **THIS GATE BLOCKS EVERY UNAFFECTED URLS OR AFFECTED URLS SECTION (AND EVERY PLANNED AFFECTED URLS SECTION).**
> The failure mode: when writing the URL sections, the page list is constructed from memory rather than from the actual filesystem. The model "remembers" `index.html` and stops there, silently omitting every other page in `live-site-pages/`. Version numbers and status indicators (🟢/🟡/🔴) also get fabricated from memory instead of read from the actual `html.version.txt` files, producing labels that contradict the real state of the repo. Descriptive advice ("enumerate before writing") does not survive task pressure — only a hard procedural gate before the URL sections does.
>
> **Relationship to the Response Opener gate.** This gate is **subordinate to the Response Opener gate and fires as part of its Step 3 (work phase)** — it runs when a URL section is about to be written, which always happens after the opener's Steps 0–2 have resolved. The "THIS GATE BLOCKS" language above applies only within that Step 3 scope; it does not compete with the Response Opener gate's "first tool call is `date`" rule. Additionally, this gate honors the Precedence Header at the top of the file: PLANNED AFFECTED URLS and the end-of-response UNAFFECTED/AFFECTED URLS sections are themselves gated by `CHAT_BOOKENDS` and `END_OF_RESPONSE_BLOCK` respectively. When the toggles suppress those sections, this gate does not fire — there is no URL section to block. Enumeration is only required when a URL section is actually going to be written.

**The hard gate — before writing ANY UNAFFECTED URLS, AFFECTED URLS, or PLANNED AFFECTED URLS section, in this exact order:**

1. **Step 1: Enumerate version-tracked pages from filesystem** — run a filesystem command to list **version files**, which transitively defines the page set. Do not list from memory. Valid command: `ls live-site-pages/html-versions/*html.version.txt 2>/dev/null` — each entry maps to a page via `<basename>html.version.txt` → `<basename>.html`. The result is the authoritative page list for this response. **Only pages with a matching `html.version.txt` are enumerated** — unversioned auxiliary HTML files in `live-site-pages/` (redirects, test stubs, archived experiments, `404.html`) are not "tracked pages" and are omitted from the URL sections. They still deploy to GitHub Pages, but they don't carry a version label and aren't part of the repo's tracked surface.
2. **Step 2: Read status + version for every tracked page** — for each version file from Step 1, read it and parse the first pipe-delimited field to determine status: empty = `🟢` Active, `maintenance` = `🟡` Maintenance, `inactive` = `🔴` Inactive. Do not infer the version or status from memory — the version file is the single source of truth. Batch the reads in parallel (or use a single `bash` loop) to minimize overhead.
3. **Step 3: Partition into affected vs unaffected, then write the sections** — for each tracked page, decide if it was affected in this response (HTML file directly edited, or its registered GAS script edited — see the GAS Projects table for the embedding-page mapping). Place affected pages in AFFECTED URLS with the post-bump version; place unaffected pages in UNAFFECTED URLS with the current version. Only after Steps 1 and 2 are complete may the URL sections be written.

**No exceptions.** Not "only one page in this repo," not "I already read versions earlier in the response." The filesystem is ground truth for every response — prior-session knowledge and mid-response memory both drift. Even on a single-page repo, Step 1 takes ~1 second and catches the case where a new page was added since the model last enumerated.

**The self-check:** before writing the first line of UNAFFECTED URLS, AFFECTED URLS, or PLANNED AFFECTED URLS, ask **two** questions — both must be YES:

1. **Enumeration** — did I read every tracked page's version file contents from the filesystem **in this response** (covering Steps 1–2 of the hard gate — list the version files, then read each one)?
2. **Partition** — did I determine which pages were affected — **HTML file directly edited** OR **registered GAS script edited per the GAS Projects table in `.claude/rules/gas-scripts.md`** — and place affected pages in AFFECTED URLS while leaving unaffected pages in UNAFFECTED URLS?

If either answer is no, STOP and run the missing step before writing. The **Partition** check is the most commonly missed — a model can correctly enumerate and read every version file, then partition incorrectly (e.g. a `.gs` edit whose embedding page was missed, so the page lands in UNAFFECTED when it should be AFFECTED). Enumeration without a verified partition produces URL sections that look complete but misrepresent what changed.

**Enumeration scope (version-tracked only):**
- Every `.html` file in `live-site-pages/` (or any `index.html` in a subdirectory) **that has a matching `html-versions/<basename>html.version.txt`**
- Unversioned HTML files (redirects, test stubs, `404.html`, archived experiments) are **excluded** — they cannot display a `(vXX.XXw)` label and are not the repo's tracked surface
- Exclude `templates/` — those are source templates, not deployed pages

**Batching with the end-of-response block preamble:** Step 1 and Step 2 can be combined with the `date` call and TODO.md read that already happen before the end-of-response block — issue all of them as parallel tool calls in the same batch. This keeps the enumeration overhead under ~5s total.

## Document-Prescribed Workflows — Closing Marker Exception

> **THIS IS AN ESCAPE HATCH** — when the "Document-Prescribed Workflows" rule in `.claude/rules/behavioral-rules.md` is active for a response (i.e. the developer pointed Claude at an implementation guide or remediation plan that prescribes pause points between steps), the standard Chat Bookends closing flow conflicts with the document's "wait for developer confirmation" requirement. Without this exception, a response that should pause after step N would close with `✅✅CODING COMPLETE✅✅` and the developer would lose the opportunity to test before step N+1.

**The rule:** when Document-Prescribed Workflows is active for this response, **the closing marker rules are suspended**. End the response with the format the document prescribes — typically `⏸️⏸️AWAITING USER RESPONSE⏸️⏸️` (if the document's pause is an `AskUserQuestion` call) or a simple text end with no closing marker at all (if the document's pause is just "describe what to test and stop"). **Do NOT emit `✅✅CODING COMPLETE✅✅`** in this case — the work is not complete, it is paused awaiting developer confirmation.

**Precedence:** Document-Prescribed Workflows **wins over** the Chat Bookends closing marker rules because the document is a session-specific override that takes precedence over the repo's default execution style (per behavioral-rules.md's "Document-Prescribed Workflows" section). When in doubt about which closes a response, check whether the developer invoked a prescribed workflow this session — if yes, use the document's format; if no, use the standard Chat Bookends closing markers.

**Recognition — when is this active?** When one or more of these apply:
- The developer's opening prompt pointed at a document (e.g. "follow the plan in `docs/migration.md`", "work through the remediation in `SECURITY-FIXES.md`")
- The document contains explicit workflow instructions ("wait for developer confirmation between steps", "implement one phase at a time, test after each", "do not advance unprompted")
- The developer has instructed mid-session that a prescribed workflow is in effect

**Scope of the suspension:** only the closing markers (`✅✅CODING COMPLETE✅✅` / `🔬🔬RESEARCH COMPLETE🔬🔬`) and the `ACTUAL TOTAL COMPLETION TIME` line are suspended. The rest of the end-of-response block (UNAFFECTED URLS, AGENTS USED, FILES CHANGED, COMMIT LOG, WORTH NOTING, SUMMARY, TODO, NEW FOLDERS, AFFECTED URLS, CLAUDE TO DEVELOPER) is still emitted when `END_OF_RESPONSE_BLOCK` = `On` — the developer still wants the work summary and forward-path recommendations; they just don't want a "done" declaration at a pause point. After the end-of-response block, emit `⏸️⏸️AWAITING USER RESPONSE⏸️⏸️` (with timestamp) instead of CODING COMPLETE, followed by the pause description from the document.

**Developer override:** if the developer says "do all remaining steps", "skip the pauses", or "just finish it" mid-session, the Document-Prescribed Workflows rule is overridden (per its own opt-out clause) — at that point the standard Chat Bookends closing flow resumes for the remainder of the session.
